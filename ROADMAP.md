## Фича 1 — Question Harvesting Pipeline (SEO-вопросы → запросы пользователей к LLM)

### Цель
Универсальный пайплайн для **любого рынка**: собрать популярные SEO-вопросы (стартуя с
Яндекс Wordstat), затем переписать их в естественные формулировки, как их реально задают
люди нейросетям. Результат — датасет реальных «промптов спроса», которым потом проверяется
видимость бренда в LLM (downstream-этап основного инструмента).

### Зачем именно переписывать
SEO-ключи — самый дешёвый прокси реального спроса, но формулируются телеграфно
(`кроссовки nike бег купить`). Запрос к ассистенту звучит иначе — разговорно, длинно, с
контекстом (`Посоветуй беговые Nike для асфальта до 12000 ₽`). Без шага переписывания мы
меряем не то, что люди реально спрашивают у ИИ.

### Стадии пайплайна
1. **Seed-вход.** Бренд + рынок/категория + конкуренты + несколько seed-ключей.
   Опционально: язык, регион, воронка (info/commercial/transactional).

2. **Расширение ключей/вопросов — слой источников (pluggable providers).**
   Чтобы пайплайн был «для любого рынка», источники прячем за единый интерфейс
   `KeywordProvider`, и рынок = выбор провайдера/локали.
   - **Яндекс Wordstat** (основной для RU). Пути доступа:
     - *Официальный Wordstat API* — `https://api.wordstat.yandex.net`, нужен OAuth-токен +
       ClientId + заявка в поддержку Yandex Direct + квоты (503 при превышении). Чистый, но
       gated путь с lead time → **подавать заявку заранее**.
     - *Сторонние агрегаторы* — Bukvarix, Key Collector, Rush Analytics, Topvisor, Mutagen.
     - *Скрейпинг* (A-Parser `SE::Yandex::WordStat` и т.п.) — против ToS, капчи, хрупко.
       Только как осознанный fallback, не для production.
   - **Для универсальности (не только RU):** Google Ads Keyword Planner API,
     Google Autocomplete/Suggest, «People Also Ask», Google Trends; Bing; реальные вопросы с
     Reddit/Quora/тематических форумов; YouTube suggest; Google Search Console запросы и логи
     поиска по сайту (если владелец даёт доступ).

3. **Фильтр вопросов-интентов.** Оставляем формулировки с вопросительным интентом:
   - RU: `как, что, какой/какая/какие, где, почему, зачем, сколько, когда, можно ли, чем,
     стоит ли, лучше, отзывы, vs/или`.
   - EN: `how, what, which, where, why, when, is/are, can, best, vs, review`.
   - Эвристика + опциональный LLM-классификатор интента (info/commercial/nav/transactional).
   - Порог по частотности (Wordstat «показов»).

4. **Кластеризация и дедуп.** Группируем близкие вопросы (эмбеддинги + кластеризация),
   выбираем каноничную формулировку на кластер, агрегируем частотность, размечаем тему и
   стадию воронки.

5. **Переписывание в LLM-промпты — ядро (LLM-шаг).**
   - Вход: SEO-вопрос + контекст бренда/рынка.
   - Выход: 1..N разговорных промптов, как их реально набрал бы человек ассистенту;
     варьируем формулировку/персону/язык, сохраняем интент, добавляем реалистичный контекст.
   - **Native structured output** (Pydantic-схема), напр.:
     `{seo_source, frequency, intent, persona, locale, prompt_text, brand_relevance}`.
   - Повторный дедуп уже по сгенерированным промптам.

6. **Выход / хранение.** Датасет `кластер → seed-вопрос → частотность → промпты → теги`.
   Питает downstream: прогон промптов через LLM и проверку, цитируется ли бренд.

### Открытые решения (гейты тех-долга)
- ⚠️ **Выбор LLM-стека — ASK FIRST (правило проекта).** Перед кодом шагов
  переписывания/классификации — подтвердить фреймворк. Идиоматичный дефолт для Python —
  `pydantic-ai` (типизированный слой, async, native structured outputs). **Не хардкодить
  bare-SDK цикл.** Требует явного подтверждения пользователя.
- Путь доступа к Wordstat: официальный API (подать заявку заранее) vs агрегаторы vs скрейпинг.
- Модель эмбеддингов + библиотека кластеризации для дедупа.
- Сколько вариантов промптов на вопрос (покрытие vs стоимость).
- Кэш-слой (квота Wordstat + стоимость LLM); prompt caching с первого дня.
- Мини eval-сет для контроля качества переписывания.

### Риски / Gotchas
- Lead time и квоты официального Wordstat API → закладывать кэш и заявку заранее.
- Скрейпинг Wordstat — ToS-риск и капчи; не тащить в production.
- Переписывание может «галлюцинировать» интент → нужен human-review / eval-петля.
- «Универсальность для любого рынка» = pluggable-источники + локале-зависимые списки
  вопросительных слов + мультиязычное переписывание (а не один захардкоженный RU-путь).

## Фича 2 — Domain GEO-Audit Gate (аудит домена как валидационный гейт)

### Цель
Быстрый автоматический аудит домена на **базовую готовность к GEO-видимости**. Запускается
**первым** при вызове основной команды как gate. Если не пройден — формируем понятный отчёт
«что добавить человеку, чтобы начать быть видимым в нейросетях».

### Важная рамка (по итогам ресёрча)
Эти файлы/сигналы — **гигиена, а не гарантированный ranking-фактор**. Сайт может уже
цитироваться в LLM через сторонние упоминания, даже не имея ничего из списка. Поэтому gate
по умолчанию **advisory** (предупреждает + объясняет), а хардблок — только на настоящих
блокерах (категория A): сайт физически недоступен для AI-краулеров. Это прямо учитывает твою
мысль «возможно он уже видим». Разделяем два вопроса: **«может ли ИИ вообще тебя прочитать»**
(жёстко) vs **«оптимизирован ли ты»** (мягко, рекомендации).

> Терминологическая поправка: файл называется **`llms.txt`**, не `llm.txt`.

### Расширенный чек-лист
Severity: 🔴 blocker (хардблок по умолчанию) · 🟡 recommended · ⚪ nice-to-have.

#### A. Crawl-доступ и протокол — настоящие блокеры
| # | Проверка | Sev | Почему важно |
|---|----------|-----|--------------|
| A1 | Домен резолвится, HTTPS валиден (TLS ок, нет mixed content), HTTP→HTTPS редирект | 🔴 | Базовый сигнал доступности и доверия |
| A2 | Homepage и ключевые страницы отдают `200` (не 4xx/5xx, без цепочек редиректов) | 🔴 | Недоступную страницу не процитируют |
| A3 | **`robots.txt` есть И НЕ блокирует AI-краулеров** (главный нюанс) | 🔴/🟡 | Блок search/retrieval-ботов = невидимость в ИИ-ответах |
| A4 | **`sitemap.xml`**: есть, валидный XML, доступен, свежий `lastmod`, указан в robots.txt | 🟡 | Помогает обнаружению и приоритизации контента |
| A5 | **Контент в «сыром» HTML (SSR), а не только в client-side JS** | 🔴 | Многие AI-краулеры не исполняют JS → не видят SPA-контент |

**Деталь A3 — каких ботов проверять в `robots.txt`** (различать политику vs самострел):
- *Training* (блок — это политика, ок): `GPTBot`, `Google-Extended`, `CCBot`,
  `Applebot-Extended`, `Amazonbot`, `Meta-ExternalAgent`, `Bytespider`.
- *Search / retrieval* (блок = убивает видимость в ИИ-ответах → 🔴): `OAI-SearchBot`,
  `PerplexityBot`, `Claude-SearchBot`, `Googlebot`, `Bingbot`, `YandexBot`.
- *User-initiated fetch* (когда пользователь сам просит ассистента сходить): `ChatGPT-User`,
  `Claude-User`, `Perplexity-User`, `DuckAssistBot`, `MistralAI-User`.
- Также проверить, что robots.txt **ссылается на sitemap**.

#### B. Машиночитаемость и структура
| # | Проверка | Sev | Почему важно |
|---|----------|-----|--------------|
| B1 | **Structured data (Schema.org JSON-LD)** есть и валиден | 🟡 | Самый высокий impact: Organization, WebSite, Product, Article/BlogPosting (author=Person, datePublished, dateModified, headline==H1), FAQPage, HowTo, BreadcrumbList. ~78% внедрений с ошибками → обязательно валидировать |
| B2 | **Семантический HTML**: контент в `<main>/<article>/<section>`, nav/aside/footer отделены, один `<h1>`, логичная иерархия заголовков | 🟡 | ИИ извлекают ответ из основного блока; ~55% цитат — из первых 30% страницы |
| B3 | Meta-база: уникальные `<title>`, description, `canonical`, `lang`, Open Graph/Twitter; ключевые страницы НЕ `noindex` | 🟡 | Идентичность страницы и отсутствие случайного исключения из индекса |
| B4 | **`llms.txt`** (+ опц. `llms-full.txt`) по `/llms.txt` | ⚪ | Emerging B2A-конвенция (~10–15% adoption, НЕ доказанный ranking-фактор). Дёшево добавить. У `llms-full.txt` следить за token-budget |

#### C. Сущность и доверие (E-E-A-T-ish, для доверия ИИ)
| # | Проверка | Sev | Почему важно |
|---|----------|-----|--------------|
| C1 | Страницы About / Contact / авторские страницы | 🟡 | Ясность сущности и авторства |
| C2 | Консистентная бренд-сущность: Organization-схема с `logo`, `sameAs` на соцсети/Wikipedia/Wikidata | 🟡 | Связывает домен с узнаваемой сущностью |
| C3 | (Опц., вне домена) присутствие в Wikipedia/Wikidata и авторитетных каталогах | ⚪ | Цитаты ИИ часто берут из сторонних авторитетных источников |

#### D. Свежесть и фиды
| # | Проверка | Sev | Почему важно |
|---|----------|-----|--------------|
| D1 | Видимые даты публикации/обновления контента | ⚪ | ИИ доверяют свежему и датированному |
| D2 | RSS/Atom-фид | ⚪ | Синдикация / удобство ингеста |
| D3 | `/.well-known/` (security.txt и будущие AI-related файлы) | ⚪ | Стандартизованные точки обнаружения |

#### E. Доп. / на будущее
- ⚪ Core Web Vitals / скорость (для GEO второстепенно, для краула полезно).
- ⚪ `ai.txt` (предложение Spawning) — emerging, опционально.
- ⚪ Favicon, корректная 404, отсутствие длинных цепочек редиректов.

### Поведение гейта и remediation-отчёт
- Запуск **первым** на основной команде. По умолчанию: блокеры (A) → хардстоп с инструкцией;
  B/C/D → предупреждения, но можно продолжить (`--force`).
- На выходе при провале — структурированный отчёт:
  - по каждой проверке: `pass / warn / fail`, severity, и **конкретное «как починить»** с
    готовым сниппетом (allow-блок в robots.txt, минимальный `llms.txt`, JSON-LD Organization).
  - единый **GEO-readiness score** + вердикт: «Готов» / «Готов с замечаниями» /
    «Критичные блокеры — почини, иначе ИИ тебя не видит».
- Кэш результатов на домен (TTL), чтобы не аудитить на каждый вызов.

### Открытые решения / тех-долг
- Политика хардблок vs warn (рекомендация: хардстоп только на категории A).
- Проверка JS-рендеринга: headless-браузер (Playwright, тяжёлая зависимость) vs лёгкая
  эвристика «есть ли основной контент в сыром HTML» — начать с эвристики.
- Валидация schema: локальная JSON-LD библиотека vs внешний валидатор.
- Глубина обхода: только homepage vs N ключевых страниц из sitemap.

### Заметка по стеку
Аудит — в основном **детерминированная HTTP/parse-работа** (`httpx` + `selectolax`/`lxml`
+ `extruct` для JSON-LD), **не LLM-агент** → правило ASK-FIRST не триггерит. Опциональный
LLM-шаг — только дружелюбно резюмировать remediation-отчёт человеческим языком.

## §3. Отложенные вопросы — решить потом

> Единый список «later». Сейчас НИЧЕГО не решаем и не делаем — просто чтобы не потерялось.
> Детали по каждому пункту — в секциях «Открытые решения» соответствующих фич выше.

### Решения перед реализацией (когда дойдут руки)
1. **LLM-стек (ASK FIRST)** — подтвердить до написания LLM-кода (Фича 1, шаги 3/5).
   Кандидат: `pydantic-ai`. Затем — skill `llm-agents`. Пока НЕ выбрано.
2. **Доступ к Wordstat** — официальный API vs агрегаторы vs скрейпинг; если официальный —
   у заявки большой lead time. Пока НЕ выбрано.
3. **Хранилище датасета вопросов/промптов** — формат и место. Пока НЕ выбрано.
4. **Где живёт gate** — как встраивается в основную команду (middleware/декоратор). Пока НЕ выбрано.
5. **Фича 1** — модель эмбеддингов + либа кластеризации; число вариантов промптов на вопрос;
   кэш-слой; eval-сет (см. «Открытые решения» Фичи 1).
6. **Фича 2** — политика хардблок vs warn; JS-рендеринг через headless vs эвристика;
   валидатор schema; глубина обхода (см. «Открытые решения» Фичи 2).

### Возможные следующие шаги (когда вернёшься к задаче)
- Разнести этот ROADMAP на отдельные файлы по фичам, если разрастётся.
- Сделать первый «спайк» по Фиче 2 (аудит почти весь детерминированный, без LLM —
  правило ASK-FIRST не триггерит).
- Оформить заявку на официальный Wordstat API (из-за lead time — имеет смысл раньше всего).
